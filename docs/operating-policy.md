# Operating Policy

## Production Data Safety

Production data is Brian's real workout system. Treat it as live personal data.

Do not mutate production data unless Brian explicitly approves the exact operation. This includes:

- inserting, updating, or deleting `sessions`
- changing production `program` or `monthly_program`
- changing `user_profile` or `user_measurements`
- changing production `daily_logs` rows or stored food photos
- broad cleanup scripts
- RLS/grant changes that could lock Brian out or expose data

Allowed without extra approval:

- read-only production checks
- code inspection
- staging-only tests
- synthetic staging users
- migration drafting

Production cutovers require explicit before/after checks and a rollback idea.

## Secrets And Auth Links

- Never commit service-role keys, access tokens, one-time magic links, or generated auth links.
- It is acceptable for the public anon key to appear in the frontend; do not treat it as a secret.
- Do not print service-role-derived artifacts in final answers.
- Remove temporary files containing auth links after QA.

## Branching

Use focused branches for risky work. Prefer `codex/...` for Codex-owned branches unless following an existing project branch.

Common branches:

- `main`: production source
- `staging`: staging source
- `gh-pages`: deployed public HTML
- `codex/...`: temporary focused Codex branches

Do not force-push, reset, or revert user work without explicit approval.
Do not develop on or hand-edit `gh-pages`; it is generated by the deploy workflows.

## Change Style

- Inspect current patterns before editing.
- Keep the static app simple unless a larger migration is explicitly planned.
- Start shared app changes in `staging` (`brian_STAGING.html`), verify them at `/staging/app/`, then port the tested change intentionally to `main/app/index.html`.
- Keep `AGENTS.md`, `APP_SOURCES.md`, and the core `docs/` operating context synchronized across `main` and `staging`.
- Add migrations for database/schema/RLS changes; do not leave live DB changes undocumented.
- Update Notion for meaningful status changes, blockers, and release notes.

## Mobile Login Known Issue

Brian often uses the app from an iPhone Home Screen web-app shortcut. Magic-link login may authenticate Safari/Chrome rather than the standalone Home Screen app session context. This is future work, not a Phase 3 blocker.

Possible future approaches:

- passkeys/WebAuthn or platform sign-in
- OTP/code entry that can be typed into the Home Screen app
- true PWA/deep-link investigation
- clearer user instructions for requesting login from the intended app context
